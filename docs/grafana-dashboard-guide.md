## 패널별 설명

### 1행: 처리량과 응답 속도

| 패널 | 의미 | 좋은 방향 | 보는 법 |
|------|------|----------|--------|
| **HTTP Request Rate (req/s)** | 초당 처리한 요청 수 | 높을수록 좋음 | 부하 테스트 중 MVC vs WebFlux 라인 차이 비교 |
| **HTTP Response Time p99 (ms)** | 100개 요청 중 가장 느린 1개의 응답 시간 | 낮을수록 좋음 | 사용자가 체감하는 "최악의 경우" |

### 2행: 응답 속도 상세

| 패널 | 의미 | 좋은 방향 | 보는 법 |
|------|------|----------|--------|
| **HTTP Response Time p95 (ms)** | 100개 요청 중 95번째로 느린 응답 시간 | 낮을수록 좋음 | p99보다 안정적인 지표, 대부분의 사용자 경험 |
| **HTTP Response Time Avg (ms)** | 전체 요청의 평균 응답 시간 | 낮을수록 좋음 | 전체적인 성능 체감 |

### 3행: 메모리와 스레드 (핵심 비교 포인트)

| 패널 | 의미 | 좋은 방향 | 보는 법 |
|------|------|----------|--------|
| **JVM Heap Memory (MB)** | 힙 메모리 사용량 | 낮을수록 좋음 | WebFlux(코루틴 수KB) vs MVC(스레드 1MB) 차이 |
| **JVM Thread Count** | 현재 활성 스레드 수 | 낮을수록 효율적 | **가장 중요한 패널** — MVC는 동시 요청 수만큼 증가, WebFlux는 거의 일정 |

### 4행: CPU와 Non-Heap

| 패널 | 의미 | 좋은 방향 | 보는 법 |
|------|------|----------|--------|
| **System CPU Usage (%)** | CPU 사용률 | 상황에 따라 다름 | MVC는 스레드 컨텍스트 스위칭으로 높을 수 있음 |
| **JVM Non-Heap Memory (MB)** | 메타스페이스, 코드 캐시 | 참고용 | 두 앱 간 큰 차이 없을 것 |

## 부하 테스트 시 예상 패턴

### 동시 200명 + 50ms delay

```
MVC 예상:
┌──────────────────────────────┐
│ Thread Count: 200개로 급증    │ ← 요청마다 스레드 1개 점유
│ Heap Memory: 높음             │ ← 스레드 스택 메모리
│ p99 Latency: 높음             │ ← 50ms delay + 스레드 풀 대기 시간
│ Request Rate: 일정 수준 후 정체│ ← 스레드 풀 한계 도달
│ CPU Usage: 높음               │ ← 스레드 컨텍스트 스위칭 비용
└──────────────────────────────┘

WebFlux 예상:
┌──────────────────────────────┐
│ Thread Count: 10~20개 유지    │ ← Netty 4개 + 보조 스레드
│ Heap Memory: 낮음             │ ← 코루틴은 수 KB
│ p99 Latency: ~50ms           │ ← delay만큼만 (스레드 풀 대기 없음)
│ Request Rate: 높게 유지        │ ← 스레드 풀 병목 없음
│ CPU Usage: 낮음               │ ← 코루틴 스위칭은 가벼움
└──────────────────────────────┘
```

### delay별 차이 예상

```
delay 0ms:
  → 차이 미미 (DB 조회 자체가 빠름, 스레드 풀 병목 없음)

delay 50ms:
  → WebFlux 우위 시작
  → MVC는 200 스레드가 50ms씩 점유 → 201번째 요청부터 대기
  → WebFlux는 코루틴 중단 → 스레드 해방 → 동시 처리 가능

delay 100ms:
  → WebFlux 압도적 우위
  → MVC 스레드 풀 고갈 → p99 급등, Error Rate 발생 가능
  → WebFlux는 여전히 안정적
```

### 동시 사용자별 차이 예상

```
50명:   MVC도 여유 (Tomcat 기본 200 스레드)
100명:  MVC 살짝 부하 시작
200명:  MVC 스레드 풀 한계 근접, WebFlux와 격차 발생
500명:  MVC 스레드 풀 초과 → 요청 대기열, WebFlux는 안정
```

## 핵심 비교 포인트 요약

| 지표 | MVC에서 나타나는 현상 | WebFlux에서 나타나는 현상 |
|------|---------------------|------------------------|
| **Thread Count** | 동시 요청 수와 비례하여 증가 | 거의 일정 (10~20개) |
| **Heap Memory** | 스레드 증가에 따라 상승 | 낮고 안정적 |
| **p99 Latency** | delay + 스레드 풀 대기 시간 | delay에 가까움 |
| **Request Rate** | 스레드 풀 한계에서 정체 | 높게 유지 |
| **CPU Usage** | 스레드 컨텍스트 스위칭으로 높음 | 코루틴 스위칭으로 낮음 |

## 사용법

### 시간 범위 조절
- 우측 상단: `Last 15 minutes` → 테스트 중 `Last 5 minutes`로 변경하면 실시간 변화 잘 보임

### application 필터
- 상단 드롭다운: `application` → `All` (둘 다 비교) 또는 개별 선택

### 자동 새로고침
- 우측 상단: `5s`로 설정되어 있어 5초마다 자동 갱신
